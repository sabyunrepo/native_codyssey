# 1. 프로젝트 개요

본 프로젝트에서는 서로 다른 자동화 도구인 Make와 Zapier를 사용하여 동일한 워크플로우를 구현하고 비교 분석하였다.

구현한 워크플로우는 RSS Feed에서 새로운 뉴스가 등록되었을 때 해당 정보를 수집한 후 Google Sheets에 저장하고 Telegram으로 알림을 전송하는 자동화 시스템이다.

두 플랫폼 모두 동일한 기능을 수행하도록 구성하였으며, 구현 과정과 사용 편의성, 확장성, 비용 구조 등을 비교하였다.

# 3. 구현한 워크플로우

## 워크플로우 구조

```text
              RSS Feed
                  │
                  ▼
        RSS 뉴스 데이터 수집
                  │
                  ▼
                Router
           ┌──────┴──────┐
           ▼             ▼
Google Sheets      Telegram
뉴스 데이터 저장     뉴스 알림 전송
```

## 동작 과정

1. RSS Feed에서 새로운 뉴스 게시글을 감지한다.
2. 뉴스 제목, 링크, 발행일 등의 정보를 추출한다.
3. Router를 이용하여 두 개의 작업으로 분기한다.
4. Google Sheets에 뉴스 정보를 저장한다.
5. Telegram으로 뉴스 알림 메시지를 전송한다.
6. 사용자는 스프레드시트에서 뉴스 이력을 관리하고 Telegram에서 실시간 알림을 받을 수 있다.

# 6. 비교 분석

## 워크플로우 화면 비교

### Make

![Make 노드](./png/make%20노드.png)

### Zapier

![Zapier 노드](./png/zapier%20노드.png)

Make는 전체 자동화 흐름을 시각적으로 한 화면에서 확인할 수 있으며, Zapier는 단계별 설정 방식으로 구성되어 있다.

---

## 비교 항목 1. UI/UX

| 항목        | Make       | Zapier     |
| --------- | ---------- | ---------- |
| 인터페이스     | 시각적 캔버스 방식 | 단계별 마법사 방식 |
| 워크플로우 가시성 | 높음         | 보통         |
| 초보자 친화성   | 보통         | 높음         |

### 분석

Zapier는 단계별로 설정할 수 있어 초보자가 사용하기 쉽다.

Make는 전체 워크플로우를 한 화면에서 볼 수 있어 복잡한 자동화 관리에 유리하다.

---

## 비교 항목 2. 설정 난이도

| 항목      | Make  | Zapier |
| ------- | ----- | ------ |
| 초기 학습   | 다소 필요 | 쉬움     |
| 단순 자동화  | 보통    | 쉬움     |
| 복잡한 자동화 | 쉬움    | 다소 어려움 |

### 분석

단순 자동화는 Zapier가 빠르게 구축 가능하였다.

복잡한 자동화에서는 Make가 더 효율적이었다.

---

## 비교 항목 3. 분기 처리 기능

| 항목    | Make   | Zapier |
| ----- | ------ | ------ |
| 분기 기능 | Router | Paths  |
| 시각화   | 우수     | 보통     |
| 확장성   | 높음     | 보통     |

### 분석

Make의 Router는 여러 개의 작업을 시각적으로 쉽게 분기할 수 있었다.

Zapier는 Paths 기능을 통해 분기가 가능하지만 시각적인 표현은 제한적이었다.

---

## 비교 항목 4. 비용 구조

| 항목       | Make      | Zapier |
| -------- | --------- | ------ |
| 과금 기준    | Operation | Task   |
| 무료 플랜    | 비교적 넉넉    | 제한적    |
| 대량 처리 비용 | 낮음        | 높음     |

### 분석

뉴스 수집량이 많아질수록 Make가 비용 측면에서 유리하였다.

Zapier는 사용량 증가에 따라 비용 부담이 커질 수 있다.

---

## 비교 항목 5. 실행 로그 및 디버깅

| 항목        | Make | Zapier |
| --------- | ---- | ------ |
| 실행 기록 조회  | 우수   | 우수     |
| 데이터 흐름 확인 | 우수   | 보통     |
| 오류 원인 분석  | 우수   | 보통     |

### 분석

Make는 각 모듈 간 데이터 이동을 시각적으로 확인할 수 있어 디버깅이 편리하였다.

Zapier도 실행 이력은 제공하지만 데이터 흐름 확인은 상대적으로 제한적이었다.

---

# 7. 각 도구의 장단점

## Make

### 장점

* Router를 통한 분기 처리 기능 제공
* 복잡한 워크플로우 구현 가능
* 데이터 가공 기능 우수
* 비용 효율성이 높음
* 시각적인 워크플로우 관리 가능

### 단점

* 초기 학습 시간이 필요함
* 초보자에게는 다소 복잡할 수 있음

---

## Zapier

### 장점

* 쉬운 설정 방식
* 빠른 구축 가능
* 다양한 SaaS 서비스 지원
* 비개발자 친화적

### 단점

* 복잡한 자동화 구현 시 제약 존재
* 비용 부담이 상대적으로 높음
* 분기 처리 기능이 제한적임

---

# 8. 결론

본 프로젝트에서는 RSS Feed에서 뉴스 정보를 수집한 후 Google Sheets에 저장하고 Telegram으로 알림을 전송하는 동일한 자동화 워크플로우를 Make와 Zapier에서 각각 구현하였다.

구현 결과 두 플랫폼 모두 요구한 기능을 정상적으로 수행하였다.

Zapier는 설정이 간단하고 빠르게 자동화를 구축할 수 있다는 장점이 있었으며, Make는 Router를 활용한 시각적인 분기 처리와 높은 확장성을 제공하였다.

특히 이번 프로젝트와 같이 하나의 데이터를 여러 서비스로 동시에 전달해야 하는 경우 Make의 Router 기능이 매우 유용하였다.

따라서 단순 자동화에는 Zapier가 적합하며, 분기 처리와 확장성이 필요한 자동화 환경에서는 Make가 더 적합하다고 판단된다.
